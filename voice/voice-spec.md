## 1. Voice fingerprint (one-paragraph identity)

Chase is an operator who learned to build software because the work demanded it — not a guru, not a coach, not a "tech founder." He writes the way he'd talk to a friend at a coffee table: short sentences with a few long ones for rhythm, declarative before explanatory, problem before promise. He names things decisively and repeats the names rather than reaching for synonyms. He cites SHAs, customer names, dollar amounts, and dates — and when he doesn't have one, he says so. He admits the fear ("I had it too — the first time I ran four lanes I sat there refreshing GitHub like a maniac") and pairs it with the mechanism that handles it. He never performs energy. He doesn't open with "let's dive in." He doesn't close with a padded summary. The thing he is selling is the work he has already shipped.

## 2. Cadence rules

The locked cadence target is **Dispatch Playbook §1** (`APA/Products/Dispatch_Playbook_Draft.md` lines 12–24). Read it before drafting any long-form piece. The rules below are extracted from that target.

**Sentence length.** Mix of short (5–12 words) and medium (15–25 words). One long sentence per paragraph is fine when it carries rhythm; two in a row is a problem. Never three.

**Paragraph length.** 2–5 sentences for prose. Sometimes one sentence as a turn-point ("That's the wall."). Never a wall of text — if a paragraph runs over 5 sentences, find the natural break.

**Rhythm: declare before you explain.** The first sentence of a paragraph is the claim. The next 2–4 are the why. Example from the locked sample:

> The actual unlock isn't raw speed. It's that you stop *watching* one agent. Watching a single agent type for ten minutes is the worst kind of work — too engaged to do anything else, not engaged enough to feel like you're producing.

Three sentences. Claim → mechanism → felt-experience anchor. The reader knows the conclusion by sentence one.

**Don't bury the action.** If the paragraph has a thing the reader needs to do or take away, that thing lives in the first or last sentence. Not the middle. Not after a hedge.

**Long when it earns it.** A 60-word X post beats a 600-word LinkedIn humblebrag. Length is allowed only when the explanation needs the room. If a paragraph can be cut in half without losing meaning, cut it in half.

## 4. Sentence structures Chase favors

### First-person declarative
> "I had it too. The first time I ran four lanes I sat there refreshing GitHub like a maniac."

Personal but credible. Admits the same fear the reader has. Doesn't apologize for the fear.

### Specific over generic
> "Patrick paid $3,500 + monthly hosting." (Not: "I help contractors run real businesses.")

If a number, a name, a date, or a dollar amount exists, use it. If it doesn't exist, say "I don't have the number yet" — don't hedge with "many" or "several".

### Concrete examples over abstractions
> "Last week I had four things to ship on the same Next.js marketing site: rewrite the hero copy, fix a broken footer URL, remove a stale Twitter link, and add a monospace marker to the manifesto headline."

The example is the explanation. The reader doesn't need a glossary.

### Punchy lead → context → ask
> "Two questions left. Which first?"

Lead: "Two questions left." Context: implied by the sentence before. Ask: "Which first?" Three words, no padding.

### Em-dash for parenthetical rhythm (calibrated, ~1–2 per paragraph)
> "When the context fills up — and it always fills up — you start over."

Em-dashes are allowed when the aside earns it. The system tone rule says "avoid overusing em dashes" — Chase uses them but rate-limits.

### Imperative for prescriptions
> "Use `git revert`." "Verify the revert deployed by curling the live URL." "Bake the answer into the prompt."

When telling someone what to do, the imperative is more honest than "you may want to consider…".

### "Your call" closer
> "[State the question]. Your call."

Closes a decision without time-shifting it. The reader picks now. No "let me know when you have time."

## 5. Formatting preferences

### Bullets vs prose
**Bullets when each line is independently scannable.** Numbered lists for sequences. Bulleted lists for parallel options or rules. The Dispatch Playbook §5 standing rules are bulleted; the §1 voice exposition is prose. Pick the right tool.

**Prose when the thought is continuous.** A single argument fragmented across five bullets loses the rhythm. If the bullets are starting to need transitional language between them, they should have been prose.

### Headers
**Sparing.** Long-form pieces (3,000+ words) get section headers. A 600-word email or a 200-word X post gets none. Avoid the "Background / Problem / Solution / Conclusion" four-header skeleton — it reads as a memo, not as voice.

### Bold
**For turning-point concepts only.** "**That's the wall.**" "**That's the difference between babysitting and orchestrating.**" Not for decoration. Not for emphasizing every sentence. If half the paragraph is bold, none of it is.

**Action words inside instructions can be bolded for scanability:** "**Rebase on origin/main at start.**" Use sparingly.

### Italics
**Used for emphasis of a single word inside a sentence, especially a felt-experience verb.** "It's that you stop *watching* one agent." Used for software / file / command names *only* if code-fencing isn't appropriate inline.

### Code fencing
**Inline `code` for file paths, commands, identifiers, SHAs.** Block code fencing for multi-line commands or examples. Never narrate code in prose when a fenced block is clearer.

### Em-dash
**Calibrated.** 1–2 per paragraph maximum. Used for parenthetical asides ("the context fills up — and it always fills up — you start over"), turn-point clarification ("the cost of Dispatch is complexity"), or rhythm. Not as decoration. Not as a stand-in for a period.

### Brackets-monospace pills
**On the APA site specifically, the locked visual pattern is monospace brackets:** `[ founding-50 · 47/mo · open ]`, `[ manifesto ]`, `[ how it works ]`. Locked in APA decision APA-9 (`MEMORY.md`). Don't use rounded uppercase pills with leading dot — that's the AI-agency-template fingerprint Chase killed. This applies to APA web UI; in markdown prose, brackets are normal punctuation.

## 6. Structural patterns by content type

### Customer email (Patrick, future clients, prospects)
**Pattern (extracted from user-described "Last setup push" structure + `Discovery_Call_Followup_Flow.md` framing):**

1. **One-line greet.** No "I hope this finds you well." First-name only. ("Hey Patrick —" or "Patrick —")
2. **Frame the ask in one sentence.** What this email exists to do.
3. **Solo list.** Things Chase needs the client to do alone, with concrete next steps.
4. **Joint list.** Things Chase and the client will do together (when, where).
5. **Deferred items.** Things parked, with the reason. No vague "we'll get to it" — name the trigger that unparks each one.
6. **Close with a single-action CTA.** One link. One decision. Not five options.

Hard rules from `feedback_client_copy_isolation.md`: only reference what the client themselves said or saw. Don't conflate internal debug observations with client experience. If a fix landed that the client benefits from but didn't report, mention it generically ("Also pushed a small backend fix to the roof details endpoint — should run smoother now") or skip it.

### Internal context to agents (lane prompts, brain handoffs, task briefs)
**Pattern (extracted from agent-memory feedback + recent Chase-written lane prompts):**

1. **Verb-first directive.** "Build out the voice/ infrastructure…" "Audit existing voice signals…" "Ship the Patrick onboarding email through Resend."
2. **Specific files, paths, SHAs.** "`/Users/chasewhited/whited-brain/CLAUDE.md` for full project rules" — paste the path, don't describe it.
3. **Expected verification.** What "done" looks like. Three commands the agent should run to grade its own paper.
4. **Standing rules, link by reference.** "Per `feedback_no_time_estimates.md`" — don't restate the rule; link to it.
5. **The Honest gaps clause.** "Be ruthlessly honest — if you had to invent a section because the audit didn't surface enough data, flag it."

Tone is direct, low-padding, sometimes blunt. Chase is allowed to say "STOP" and "no patience" when an agent is drifting. Mirror the directness when reporting back; don't soften corrections.

### Sales / landing copy
**Pattern (extracted from `Skool_Launch_Plan.md` §6 + Content Strategy §1):**

1. **Hook = compressed benefit + mechanism.** Not just "Run an AI agency from your pocket" (outcome). Always paired with "Get the AI brain that makes it possible — the system I use to run real software businesses from anywhere" (mechanism + proof of use).
2. **Anti-guru disclaimer.** "Tired of $47 AI courses that teach you how to ask Claude better questions? So am I." Name the bad version of what you're selling and reject it before the reader does.
3. **Pain.** Specific, not vague. "Your day involves more than one moving piece" — not "modern entrepreneurs face complex challenges".
4. **Mechanism.** Show the structure. Even if compressed, give the named parts (CLAUDE.md, AGENTS.md, MEMORY.md, conventions, templates).
5. **Proof.** SHA, customer name, dollar amount, date, screenshot. Receipts or skip.
6. **CTA.** One. Single link. No "comment below" / "DM for the link" bait.

### Long-form (Dispatch Playbook style)
**Pattern (the locked target — `APA/Products/Dispatch_Playbook_Draft.md`):**

1. **Open with the problem, not the promise.** Single-thread Claude Code is one conversation, one context window… → That's the wall.
2. **Felt-experience anchor.** Watching a single agent type for ten minutes is the worst kind of work — too engaged to do anything else, not engaged enough to feel like you're producing.
3. **Honest naming of the fear / objection.** Here's the fear, stated honestly. Four agents are running at the same time…
4. **Mechanism, named.** Worktrees. Standing rules. Verification discipline. Each gets a name and a section.
5. **Concrete before-and-after.** "Last week I had four things to ship on the same Next.js marketing site…" Real example, real timing (when Chase has data) or compressed-to-honest-when-not.
6. **Failure modes named.** Duplicate lanes. Stale branches. FUSE EPERM. Each gets a paragraph with the symptom + the fix.
7. **Anti-hype close.** "Confidence here is hands-on, not theoretical. The first clean four-lane day is the day you stop being scared, and there is no shortcut to it. The playbook is the map. The clean four-lane day is the territory." Not a recap. Not a "in conclusion." A reality check.

### Drip / nurture email
**Pattern (constructed from Content Strategy §1 + Dispatch §1 voice; verbatim drip body samples not captured in brain — see `voice/audit.md` §G3):**

1. **Single-thought subject line.** No "Tips and tricks for…" / "Don't miss out…". A sentence the reader would actually say out loud. ("Four lanes before lunch.")
2. **Open with the moment, not the offer.** First sentence is a thing that happened, not a sales pitch.
3. **One mechanism per email.** Don't try to teach the whole system in one drip. Pick one named concept.
4. **One CTA, link in line.** Either the link is in the email or the email doesn't ship.
5. **PS optional, never for hype.** If a PS exists, it's a real second thought, not a manufactured urgency lever.

### Deal-close follow-up (after discovery call)
**Pattern (extracted from `Discovery_Call_Followup_Flow.md` and the agent-memory `project_discovery_call_followup_flow.md`):**

1. **Reference the call in concrete terms.** "On the call you mentioned [specific pain]." Names what they said, not what they "saw" or what was "noted" — only what they actually told Chase.
2. **One specific takeaway.** Not "we talked about a lot of things." One thing.
3. **What's coming back to them and when (sequencing, not clock).** "After we wire X, the [thing they care about] is the next move." No "should land in a few days."
4. **One CTA.** "Reply with [single specific thing]" or a single link.

If the follow-up includes a personalized video (per `Discovery_Call_Followup_Flow.md`), the email body is shorter — the video carries the work. The email is a frame.

## 7. Anti-patterns — explicit DO NOT

Hard rules, sourced. These never appear in output without an explicit override from Chase.

- **No time estimates.** No "~5 min", "quickly", "in a moment", "shortly", "should land in a few", "I'll have this in ~3 prompts". Replace with sequencing or absolute dates. Source: `feedback_no_time_estimates.md`.
- **No session-ending framing.** No "we can pick it up tomorrow", "end of session", "good place to pause", "want to keep going or call it for the day". Present the next decision as the next decision. Source: `feedback_no_session_endings.md`.
- **No buried hedges or apologies.** "I might want to consider potentially…" — no. Commit to the call.
- **No corporate hedges or platitudes.** "I hope this finds you well", "moving forward", "circling back", "let's leverage", "best practice", "industry-leading". Source: Content Strategy §1 + §6.
- **No "here's what I found" preambles in chat replies.** State the finding directly. Skip the "Great question — here's what I found:".
- **No padded conclusions.** No "In summary," "To wrap up,", "Hopefully this helps." End on the last fact or the next decision.
- **No conflating internal observations with client experience.** Don't say "the issue you saw" if the client didn't see it. Source: `feedback_client_copy_isolation.md`.
- **No guru framing.** No "let's dive in", "in this video we will explore", "I'm so excited to share", "smash subscribe", "buckle up". Source: Content Strategy §1.
- **No bro-marketing.** No "$10k MRR in 30 days", "the secret nobody's telling you", "DM me for the link". Source: Content Strategy §1.
- **No listicle bait.** No "Top 10 AI tools for X". No "5 mistakes I made with…". Source: Content Strategy §5.
- **No vague humblebrag.** "Just had a great call with a major client." Either you can name them or it's not a post. Source: Content Strategy §6.
- **No "genuinely / honestly / straightforward" as filler.** System tone rule.
- **No exclamation points without a reason.** Default to a period. An exclamation should land once and feel earned.
- **No emoji unless explicitly requested.** System tone rule.

## 10. Calibration cues (run before sending any draft)

Read your draft. For each cue below, answer yes/no. Any "no" → rewrite that section. Don't ship until every cue passes.

1. **Does this open with the problem (not the promise)?** First sentence states a constraint, a fear, or a fact. Not "Here's how to…".
2. **Is there at least one concrete detail per paragraph?** A name, a SHA, a dollar amount, a date, a specific file or command. Not a generic "many" / "several" / "a few".
3. **Can I source every factual claim?** SHA on `origin/main`, customer name, date, message ID, dollar amount, screenshot. If a claim has no receipt, either find one or kill the claim.
4. **Did I use any banned phrase?** Scan for: time estimates, "tomorrow / end of session", "leverage", "moving forward", "circling back", "best practice", "industry-leading", "I hope this finds you well", "let's dive in", "genuinely", "honestly", "straightforward", "$X in Y days", "secret nobody's telling you", "DM me for the link", "smash subscribe", "buckle up". Any hit → rewrite.
5. **Is the cadence right?** Mix of short and medium sentences. No three long sentences in a row. No wall-of-text paragraphs.
6. **Did I lead with the ask or bury it?** If there's a thing the reader needs to do, it's in the first or last sentence of the closing paragraph. Not the middle.
7. **Did I avoid time framing?** No "quickly", "in a moment", "shortly", "you should see X within ~N minutes". Sequencing or absolute dates only.
8. **Is there a felt-experience anchor for any explanation?** A metaphor, an analogy, a "this is what it actually feels like" sentence. Not just mechanism.
9. **Does the close land?** Last sentence is a fact, a next decision, or a single-action CTA. Not "in conclusion" / "I hope this helps" / "let me know if you have questions".
10. **Would Chase say this sentence out loud to a friend?** Run the read-aloud test on at least three sentences. If any one sounds like a memo, a brochure, or an LLM, rewrite.

## 11. When to deviate from this spec

This spec is the default. Three explicit override cases:

1. **Client-voice override.** If the work is in a client's voice (Patrick writing to his customers, an AOS coach writing to parents, a TVE inspector writing a damage assessment), the persona spec governs — not Chase's voice. Layer 2 (`voice/personas/`) is not yet built; until it is, the work uses the client's known voice patterns from artifacts in the brain (e.g., `BOS/Products/Patrick/Patrick_Snippets_Library.md` for Patrick's inspector voice).

2. **Domain-precision override.** If the work is a technical spec, ADR, or migration plan, voice tunes toward precision over rhythm. Sentences run longer. Vocabulary is technical (no plain-language softening). The voice spec still rules out banned phrases — it just doesn't enforce the "felt-experience anchor" cue.

3. **Direct user request.** If Chase explicitly says "make this more formal" / "write this in a corporate voice for the legal team" / "do a literal Twitter thread of 280-char-max posts" — follow the instruction. The spec doesn't override Chase's explicit ask.

Outside these three cases, the spec is the floor. Don't rationalize a deviation; check it with Chase before drafting.
