# memory-conventions/ — ready-to-paste brain conventions

Each file in this directory is a **feedback-type memory entry** that codifies a brain-operating convention. Copy any of them into your own brain's `memory/` directory and the convention starts applying.

These conventions are battle-tested patterns from running real multi-agent, multi-lane operations. They solve problems you will hit eventually:

- **`supersession-pattern.md`** — when a decision or fact gets replaced, never delete the old version. Mark it superseded and leave a forward pointer. Preserves the audit chain so future agents can answer "why did this change?"
- **`cascade-staleness.md`** — when you correctly update an upstream fact, downstream docs that derived from it silently drift. Pattern: docs declare `**Depends on:**` blocks; `bin/stale-audit.sh` surfaces dependents whose upstreams just changed.
- **`lane-current-state.md`** — each business lane gets an auto-generated `Current_State.md` rolled up from its Decision Log + Daily Log + Feature Inventory. Single-file cold-start handoff. Generated by `bin/lane-summary.sh`; never hand-edited.
- **`four-place-rule.md`** — every meaningful change lands in four files: Daily Log, Change Log, Decision Log, Feature Inventory. If a change isn't in all four places, it isn't shipped.
- **`cloudflare-token-verify-gotcha.md`** — account-scoped Cloudflare tokens verify at a different endpoint than user-scoped ones. The first endpoint returns "Invalid" for valid tokens.
- **`vercel-sensitive-env-gotcha.md`** — Vercel env vars marked Sensitive are unreadable via the CLI/API. Don't conclude they're missing — verify via runtime smoke test.

---

## How to apply

Copy any `.md` file in this directory into your brain's `memory/` directory. Add a one-line pointer to it in your `MEMORY.md` index:

```markdown
- [supersession-pattern.md](memory/supersession-pattern.md) — never delete superseded entries; mark and forward-point
```

That's it. The convention is now part of your brain's standing rules.

You can also modify the file before pasting — these are starting points. The "Why" sections are tied to general operating logic, not anyone's specific business. Keep them honest to your own context.

---

## Where these came from

These are the conventions that emerged from operating a real multi-agent brain across multiple products and client builds. They have been scrubbed of any operator-specific identifiers, client names, and credentials so they drop in clean to any brain.

If you discover a new convention that's worth codifying in this directory, the bar is: it solved a real, repeated problem; it's stack-agnostic; it can be applied by a coding agent without further explanation.
