# Per-lane Current_State.md — single-file cold-start handoff

Each business lane in the brain — a top-level subdirectory that represents a product, business line, or workstream — gets a `Current_State.md` file at the lane root, auto-regenerated on demand from that lane's Decision Log + Daily Log + Feature Inventory.

**Why:** Cold-start handoff overhead is the dominant cost of multi-lane work. A cross-lane "ecosystem status" doc handles the wide view well, but it's tuned for "what's happening across the agency this week," not "what does this specific lane look like right now." A new agent dropped into one lane has to read the full Decision Log (often 300+ lines, growing daily) + Daily Log (250+ lines) + Feature Inventory (100+ lines) before being safe to act. Per-lane `Current_State.md` collapses that to a single file with the load-bearing ~80% of the context.

The doc is a *derived view*. Source-of-truth always lives in the Decision Log + Daily Log + Feature Inventory. `Current_State.md` is regenerated from them; never the other way around.

## How to apply

**Generation:** `bash bin/lane-summary.sh <lane>` reads the lane's source files and writes `<lane>/Current_State.md`. Lane name is the directory name.

**What it pulls:**

- Top 5 most recent Decision Log entries (newest first), with date, title, and decision summary line
- Top 3 most recent Daily Log entries (newest first), with date and headline
- Top 5 open questions from the Feature Inventory's `Open Questions / Onboarding Deltas` section (or equivalent — the script greps for `## Open Questions` style headers)
- Header block: lane name, last regenerated date, source file list, regen command

**Naming conventions handled:** the script tries both `<Lane>/Decision_Log.md` and `<Lane>/<Lane>_Decision_Log.md` (some lanes use the prefixed form, others don't). Missing source files are noted in the output rather than failing the script.

**Cadence:** regenerate weekly per lane, or on-demand whenever there's a major lock (3+ new decisions in a day, a phase boundary, a positioning shift). The script is idempotent — re-runs overwrite cleanly.

**Never hand-edit.** If you want to add something, add it to the source (Decision Log entry, Daily Log entry, Feature Inventory open question) and the next regen picks it up. Hand-edits to `Current_State.md` are silently overwritten on the next run, which is the right behavior — the file is a view, not a source.

## Header format

Every regenerated file starts with:

```
# <Lane> — Current State

**Last regenerated:** YYYY-MM-DD HH:MM (script: `bin/lane-summary.sh <lane>`)
**Source files:**
- <lane>/Decision_Log.md (or <lane>_Decision_Log.md)
- <lane>/Daily_Log.md (or <lane>_Daily_Log.md)
- <lane>/Feature_Inventory.md

> This file is **auto-generated**. Do not hand-edit — edits are overwritten on next regen. To change content here, edit the source file and re-run the script.
```

The `Last regenerated` line + `Source files` list are the traceability backbone. Any agent reading the file can see when it was generated, what it was generated from, and how to regenerate it.

## Don't apply to

- Brain-root files (no single owning lane)
- Sub-product directories inside a lane — those have their own per-product Feature Inventories already; rolling them up into a sub-product Current_State is overkill at current volume
- One-shot project repos
- Lanes without all three source files (the script will skip them with a warning)

## Tooling

- Script: `bin/lane-summary.sh <lane>` (mode 755)
- Output path: `<lane>/Current_State.md`
- Idempotent (re-runs overwrite previous Current_State.md cleanly)
- Run cadence: weekly per lane, or on-demand after a major lock day
- Failure mode: if a source file is missing, the script notes it in the output rather than aborting (so a partially-formed lane still gets a useful summary)

## Dashboard surfacing

If you wire a dashboard on top of your brain, `Current_State.md` files at lane roots are usually picked up by any tree walker that already scans `<lane>/*.md`. No additional config needed in most setups.
